# Internationalization (i18n) Design

## Goal

Add language switching to Meridian with a community-friendly translation workflow. Anyone can contribute a translation via a PR. Power users can install custom translations locally without rebuilding the app.

---

## Library: i18next + react-i18next

**Why i18next:**
- Industry standard for React apps
- Known format — translators worldwide already know it
- Supports plurals, interpolation, fallback languages
- Pluggable backends (load from disk, remote URL, etc.)
- `react-i18next` gives `useTranslation()` hook and `<Trans>` component

**Install:**
```bash
npm install i18next react-i18next
```

No other dependencies needed for the core setup.

## File Structure

```
meridian/src/renderer/src/i18n/
  index.ts              # i18next initialization
  locales/
    en.json             # English — source of truth

    ...

# User-installed translations (outside app bundle):
~/Library/Application Support/meridian/locales/

  pt-BR.json            # Brazilian Portuguese (user installed)
```

## Architecture

### `src/renderer/src/i18n/index.ts`

```ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import en from './locales/en.json'
import ru from './locales/ru.json'

const BUNDLED_LOCALES: Record<string, object> = { en, ru }

export async function initI18n(language: string) {
  // Load user-installed locales from disk via IPC
  const userLocales = await loadUserLocales()

  const resources: Record<string, { translation: object }> = {}
  for (const [lang, msgs] of Object.entries({ ...BUNDLED_LOCALES, ...userLocales })) {
    resources[lang] = { translation: msgs }
  }

  await i18n.use(initReactI18next).init({
    resources,
    lng: language,
    fallbackLng: 'en',
    interpolation: { escapeValue: false },
  })
}

async function loadUserLocales(): Promise<Record<string, object>> {
  try {
    const files = await (window.vault as any).listUserLocales?.() ?? []
    const result: Record<string, object> = {}
    for (const { lang, content } of files) {
      result[lang] = content
    }
    return result
  } catch {
    return {}
  }
}

export { i18n }
export const availableLanguages = (localeFiles: string[]) =>
  ['en', 'ru', ...localeFiles].filter((v, i, a) => a.indexOf(v) === i)
```

### Usage in components

```tsx
import { useTranslation } from 'react-i18next'

function StatusBar() {
  const { t } = useTranslation()

  return (
    <div>
      <span>{t('statusBar.words', { count: wordCount })}</span>
      {cursorPos && <span>{t('statusBar.line', { line: cursorPos.line, col: cursorPos.col })}</span>}
    </div>
  )
}
```

### Language setting

In `useSettingsStore.ts`, add:
```ts
language: string  // 'en', 'ru', 'de', etc. — default: 'en'
```

In Settings → Appearance, add a language dropdown that calls:
```ts
i18n.changeLanguage(newLang)
s.updateSetting('language', newLang)
```

In `App.tsx` on startup:
```ts
const language = useSettingsStore.getState().language
await initI18n(language)
```

## IPC for User-Installed Locales

### New IPC: `locale:list-user`

**Main process** — scan `{userData}/locales/*.json`:
```ts
ipcMain.handle(IPC.LOCALE_LIST_USER, async () => {
  const dir = join(app.getPath('userData'), 'locales')
  if (!existsSync(dir)) return []
  const files = readdirSync(dir).filter(f => f.endsWith('.json'))
  return files.map(f => ({
    lang: f.replace('.json', ''),
    content: JSON.parse(readFileSync(join(dir, f), 'utf-8'))
  }))
})
```

Users drop their `ja.json` into `~/Library/Application Support/meridian/locales/` and restart — Japanese appears in the language picker automatically.

## Community Workflow

### For contributors

1. Fork the repo on GitHub
2. Copy `src/renderer/src/i18n/locales/en.json` to `xx.json` (ISO 639-1 code)
3. Translate all values (keys stay in English)
4. Submit PR with title: `i18n: add [Language] translation`

### `TRANSLATING.md` (in repo root)

Guides contributors on:
- Which file to copy
- How to test locally (change language in Settings)
- Placeholder syntax (`{{count}}`, `{{name}}`)
- How to handle plurals per language
- Which keys can be left as-is (e.g. keyboard shortcuts)

### Keys that should NOT be translated

- Keyboard shortcut strings (`⌘K`, `⌘S`, etc.)
- File extensions (`.md`, `.canvas`)
- Technical terms where English is standard (`Markdown`, `Git`, `GitHub`)
- URLs

Mark these with a comment convention in `en.json` or just document in `TRANSLATING.md`.

## What NOT to do

- **Don't use gettext/PO files** — less familiar to JS devs, harder to contribute to
- **Don't auto-translate** — machine translations are often wrong for UI strings, confuse users
- **Don't namespace per file** — for this app size, one flat JSON is simpler and easier to translate
- **Don't load translations from a remote URL** — adds latency, network dependency, complexity; GitHub PRs are enough
